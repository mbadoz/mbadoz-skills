# Domain Skill Template

When generating a new domain skill, create these files exactly.
Replace all `{{placeholders}}` with values from the discovery interview.

## Directory Structure

```
{{skill-name}}/
├── SKILL.md
└── references/
    ├── domain-profile.md
    ├── principles.md
    └── learning-log.md
```

---

## File 1: SKILL.md

```markdown
---
name: {{skill-name}}
description: |
  Learned principles for {{domain description}}.
  Use this skill when {{trigger conditions}}.
  Consult references/principles.md before producing any deliverable.
  This skill was generated by the adaptive-learner meta-skill.
---

# {{Skill Title}}

## Before Producing a Deliverable

1. Read `references/principles.md` — apply all principles with
   confidence >= 3 stars
2. Prioritize by category order (defined in domain-profile.md)
3. If a principle conflicts with the current context, flag it
   to the user
4. After delivery, ask: "Do you have feedback to help me
   improve for next time?"

## When Receiving Feedback

Follow the adaptive-learner methodology:

1. Read `references/domain-profile.md` for analysis framework
2. Extract all raw observations from feedback
3. Apply the 3-level filter to each observation:

### Filter 1 — Contextual?
> "Would this change if the product/subject changed?"
> YES --> REJECT | NO --> Continue

### Filter 2 — Generalizable?
> "Can this be expressed as a rule WITHOUT referencing
> specific content?"
> NO --> REJECT | YES --> Continue

### Filter 3 — Structural?
> "Is this about HOW (structure, timing, hierarchy) rather
> than WHAT (subject, color, specific image)?"
> NO --> REJECT | YES --> ACCEPT

4. For each accepted principle, formulate BOTH levels:
   - **Conceptual**: The rule in plain language
   - **Implementation**: The concrete code in the tech stack
   - **Anti-pattern**: What NOT to do (if applicable)
5. Update `references/principles.md` with new/reinforced principles
6. Log the session in `references/learning-log.md`
```

```markdown
## Success Criteria
- **Metrics**: {{how success is measured}}
- **Judge**: {{who evaluates — user, metrics, both}}
- **Feedback format**: {{how feedback is provided}}

## Analysis Categories (by priority)
1. {{category 1 — highest priority}}
2. {{category 2}}
3. {{category 3}}
4. {{category 4}}
5. {{category 5}}
...

## Analysis Methodology (MANDATORY)

For each observation during feedback analysis, apply these
3 filters IN ORDER:

### Filter 1 — Contextual?
> "Would this change if the product/subject changed?"
- YES --> REJECT (e.g., "red works well" depends on product)
- NO  --> Continue to Filter 2

### Filter 2 — Generalizable?
> "Can this be expressed as a rule WITHOUT referencing
>  specific content?"
- NO  --> REJECT (e.g., "use airplane image" is too specific)
- YES --> Continue to Filter 3

### Filter 3 — Structural?
> "Is this about HOW (structure, timing, hierarchy) rather
>  than WHAT (subject, color, specific image)?"
- NO  --> REJECT
- YES --> ACCEPT as principle

### After validation:
1. Formulate the conceptual level (plain language rule)
2. Translate to implementation level (concrete code in tech stack)
3. Document the anti-pattern (what NOT to do)
4. Evaluate confidence (how many examples confirm vs contradict)
```
